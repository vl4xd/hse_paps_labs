## 2. Перечень функциональных требований

### User story mapping (USM)

USM рассмотрен для пользователей типа: "Тренер", "Фанат". Данные типы пользователей имеют идентичный функционал.

1. **Переходит на сайт**
    1. Главная страница сайта должна демонстрировать информацию о продукте
    2. Главная страница сайта должна содержать обучение по работе с сайтом
    3. Главная страница сайта должна содержать статистику о кол-ве верно/неверно данных прогнозов
    4. Гланвая страница сайта должна содержать контактную информацию
2. **Выбирает раздел спортивных состязаний**
    1. Раздел должен содержать спортивное состязание "Футбол"
3. **Выбирает подраздел спортивного состязания**
    1. Раздел "Футбол" должен содержать подраздел "Российская Премьер-Лига"
4. **Просматривает список предстоящих событий**
    1. Список должен обладать функцией отбора событий по выбранному временному периоду (дд.мм.гггг - дд.мм.гггг)
    2. Список должен обладать функцией поиска по названию команды
    3. Страница предстоящих событий должна обладать функцией пагинации
    4. Система должна запускать событие на сбор данных каждый день в определенное время
    5. Система должна производить сбор данных (признаки множетсва для модели) предстоящих событий
    6. Система должна заносить собранные данные в БД
    7. Список должен выводить для каждого события краткий прогноз (Команда 1: Победа %, Ничья %, Команда 2: Победа %) по собранным данным
5. **Выбирает предстоящее событие из списка**
    1. Предстоящее событие должно предоставлять информацию о команде
    2. Предстоящее событие должно содержать прогноз выйгрыша для каждой команды и прогноз - ничья
    3.  Система должна делать прогноз победы для каждой команды и прогноз - ничья
    4.  Система должна выводить полезные рекомендации по увеличению прогноза выйгрыша для команд путем фиксации одинх входных параметров и изменения других входных параметров
6. **Просматривает список активных событий**
    1. Список должен обладать функцией отбора событий по выбранному временному периоду (дд.мм.гггг - дд.мм.гггг)
    2. Список должен обладать функцией поиска по названию команды
    3. Страница активных событий должна обладать функцией пагинации
    4. Система должна производить сбор данных (признаки множества для модели) активных событий по ходу матча
    5. Система должна заносить собранные данные в БД
    6. Список должен выводить для каждого события краткий прогноз (Команда 1: Победа %, Ничья %, Команда 2: Победа %) по собранным данным
7. **Выбирает активное событие из списка**
    1. Активное событие должно предоставлять информацию о команде
    2. Активное событие должно содержать прогноз выйгрыша для каждой команды и прогноз - ничья
    3. Система должна делать прогноз победы для каждой команды и прогноз - ничья по ходу события
    4. Система должна выводить полезные рекомендации по увеличению прогноза выйгрыша для команд путем фиксации одинх входных параметров и изменения других входных параметров
8. **Просматривает список прошедших событий**
    1. Список должен обладать функцией отбора событий по выбранному временному периоду (дд.мм.гггг - дд.мм.гггг)
    2. Список должен обладать функцией поиска по названию команды
    3. Страница прошедших событий должна обладать функцией пагинации
    4. Система должна заносить и (или) изменять данные в БД по окончанию события
    5. Система должна обновить (дообучить) модель данными прошедшего события
9. **Выбирает прошедшее событие из списка**
    1. Прошедшее событие должно предоставлять информацию о команде
    2. Прошедшее событие должно содержать информацию о сделанных системой прогнозах "до" и "по ходу" события

## 5. Перечень нефункциональных требований

### Эксплуатационные свойства архитектуры

|Свойство|Описание|Решение|
|---|---|---|
|Доступность|Система прогнозирования должна быть доступна круглосуточно, поскольку события могут происходить в любое время|Использование резервных серверов и переход на них в случае сбоя|
|Бесперебойность|Система должна восстанавливаться в течении 24 часов после сбоя|Ручное решение сбоев|
|Производительность|Система должна справляться с пиковыми нагрузками, особенно во время крупных спортивных событий|Проведение стресс-тестирования, оптимизация кода для минимизации времени отклика (до 300-400 мс), использование капчи при подозрительной активности пользователя|
|Мастштабируемость|Возможность добавлять в систему прогнозирования без внесения больших изменений новые разделы и подразделы событий|Реализация системы на базе распределенной архитектуры - архитектуры на основе сервисов|

### Структурные свойства архитектуры
|Свойство|Описание|Решение|
|---|---|---|
|Переносимость|Система должна запускаться на настольной и мобильной версии популярных браузеров (Google Chrome, Yandex.Browser, Mobile Safari)|Оптимизация и тестирование системы для отображения в вышеописанных браузерах|
|Поддерживаемость|Система должна осуществлять техническую поддержку через отправку пользователем писем на почту|Регистрация почты для поддержки системы, указание адреса почты на главной странице системы, регулярные, ручные или автоматизированные, просмотр и ответы на письма пользователей|
